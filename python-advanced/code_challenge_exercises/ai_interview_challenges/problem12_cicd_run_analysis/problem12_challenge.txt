PIPELINE RUN ANALYSIS — JSONL PROCESSING

PURPOSE
Write one function that reads a pipeline_runs.jsonl file and returns a fixed summary dictionary.

This specification is intended to match the provided solution structure and behavior exactly.

CONSTRAINTS
- Only use Python standard library
- You may use json
- No external libraries
- Use dictionaries, loops, if/else, try/except
- Skip malformed rows safely

========================================================
1) FUNCTION SIGNATURE
--------------------------------------------------------
Implement exactly this function:

    analyze_pipeline_runs(file_path: str, failure_threshold: int = 10) -> dict

Do not add extra parameters.
Do not print anything.
Return one dictionary.

========================================================
2) REQUIRED RETURN OBJECT
--------------------------------------------------------
Return exactly this structure:

{
    "pipeline_counts": {},
    "status_counts": {},
    "top_failed_pipeline": None,
    "frequent_failure_pipelines": {},
    "busiest_minute": None,
    "average_duration_success": None,
}

These six keys must always exist in the returned dictionary,
even if the file is missing or all rows are invalid.

========================================================
3) HELPER STATE INITIALIZATION
--------------------------------------------------------
Initialize the following:

pipeline_failure_counts = {}
minute_counts = {}
success_duration_total = 0
success_count = 0

========================================================
4) FILE HANDLING RULE
--------------------------------------------------------
Open the file using a context manager inside a try block.

If opening the file raises FileNotFoundError:
return the empty results structure immediately.

Do not raise the error further.

========================================================
5) LINE PROCESSING RULES
--------------------------------------------------------
Process the file line by line.

For each line:
1. Strip whitespace
2. Skip it if it becomes empty

Then:
- Use a try block to call json.loads(line)
- Store the parsed result in data
- If json.loads raises a ValueError:
    - skip the row using continue

========================================================
6) REQUIRED FIELD VALIDATION
--------------------------------------------------------
For each parsed record:

A) Read values into:
- pipeline
- status
- duration
- timestamp

B) Required value check
- If any of pipeline, status, or timestamp is missing or empty,
  skip the row immediately

C) Count updates happen before timestamp parsing
After the required value check passes:

1. Update results["pipeline_counts"]
- If pipeline is not already a key, initialize it to 0
- Increment the count for pipeline

2. Update results["status_counts"]
- If status is not already a key, initialize it to 0
- Increment the count for status

D) Timestamp validation for minute tracking
- Check whether "T" is in timestamp
- If not, skip the row

- Split timestamp into date_part and time_part using "T"
- Check whether ":" is in time_part
- If not, skip the row

- Split time_part by ":" and store the result in time_parts
- If time_parts has fewer than 2 elements, skip the row

- From time_parts:
    - first value → hour
    - second value → minute
- Create minute_key as a string in this format:

    YYYY-MM-DD HH:MM

- Use date_part directly when building minute_key

E) Minute count update
- If minute_key is not already a key in minute_counts, initialize it to 0
- Increment the count for minute_key

F) Failure tracking
- Check whether status is exactly "failed"
- If it is:
    - If pipeline is not already a key in pipeline_failure_counts,
      initialize it to 0
    - Increment the count for pipeline

G) Success duration tracking
- Check whether status is exactly "success"
- If it is:
    - Check whether duration is an int or float
    - Only if it is an int or float:
        - add duration to success_duration_total
        - increment success_count

========================================================
7) FINAL COMPUTATION RULES
--------------------------------------------------------
These happen after the main file loop has finished.
These do not happen inside the main file for loop.

A) Compute top_failed_pipeline
- Check whether pipeline_failure_counts is not empty
- If it is not empty:
    - Find the highest count value in pipeline_failure_counts.values()
      and store it in highest_failure_count
    - Create an empty list named failure_candidates
    - Loop through each pipeline_name in pipeline_failure_counts
    - If the count for pipeline_name is equal to highest_failure_count:
        - append pipeline_name to failure_candidates
    - Assign the smallest value in failure_candidates
      using standard string comparison to results["top_failed_pipeline"]

B) Compute frequent_failure_pipelines
- Loop through each pipeline_name and count in pipeline_failure_counts.items()
- If count is greater than or equal to failure_threshold:
    - copy that pipeline_name and its count into results["frequent_failure_pipelines"]
- Do not sort anything

C) Compute busiest_minute
- Check whether minute_counts is not empty
- If it is not empty:
    - Find the highest count value in minute_counts.values()
      and store it in highest_minute_count

    - Create an empty list named minute_candidates
    - Loop through each current_minute in minute_counts
    - If the count for current_minute is equal to highest_minute_count:
        - append current_minute to minute_candidates
    - Assign the smallest value (min()) in minute_candidates
      using standard string comparison to results["busiest_minute"]

D) Compute average_duration_success
- Check whether success_count is greater than 0
- If it is:
    - Divide success_duration_total by success_count
    - Store the result in average_duration
    - Round average_duration to 2 decimal places
    - Assign the rounded value to results["average_duration_success"]

========================================================
8) RETURN RULE
--------------------------------------------------------
This happens after the main file loop and after all final computations.
Return results.
